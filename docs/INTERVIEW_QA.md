# Price Tracker 面试问答

## 1. 这个项目解决什么问题？

这是一个商品价格跟踪后端项目。用户可以注册登录、创建商品、关注商品并设置目标价。系统刷新商品价格后，如果价格达到目标价，就通过 RabbitMQ 异步创建站内通知。项目重点展示完整后端业务链路、缓存设计、MQ 解耦、幂等处理和端到端验证。

## 2. 为什么使用 RabbitMQ？

价格刷新和通知落库是两个不同关注点。刷新价格是主流程，通知处理可能涉及校验、落库、后续扩展邮件短信等。如果直接同步处理，通知链路异常或变慢会影响刷新接口和定时任务。RabbitMQ 用来解耦这两个过程，让刷新逻辑只负责投递消息，Consumer 异步完成通知落库。

## 3. 为什么使用 Redis？

Redis 在项目中不只是缓存，还承担高并发辅助能力。商品详情和当前价格用 Redis 缓存降低 MySQL 压力；空值缓存处理穿透；分布式锁处理热点 key 击穿；TTL 随机偏移降低雪崩风险；关注列表缓存减少重复查询；通知发送和 MQ 消费用 Redis 做短期幂等；关注写接口用 Redis 做限流。

## 4. 如何避免重复通知？

项目有三层防重。第一层是发送前的 Redis 入口幂等，key 基于 `userId:productId:targetPrice`，默认 TTL 10 分钟。第二层是 Consumer 消费幂等，优先使用 `messageId` 生成 Redis key，重复消息直接跳过。第三层是业务防重，`tb_watchlist.last_notified_price` 保存最近通知价格，如果同一价格已经提醒过，就不再插入通知。

## 5. MQ 消费失败怎么办？

当前实现中，`PriceAlertConsumer` 会捕获异常并记录 `Notification send failed`，不会重新抛出异常，因此不会触发 broker 重投。这个设计适合当前学习项目的主流程保护，但生产环境应该补充死信队列、重试策略、异常告警和可观测面板。

## 6. Redis 和 MySQL 如何保证一致性？

当前项目采用旁路缓存模式。读商品详情或价格时先查 Redis，未命中再查 MySQL 并回填缓存。商品更新、删除和价格刷新后，会删除商品详情缓存、价格缓存和对应空值缓存，让下一次读取回源 MySQL。这个方案保证最终一致性，不做强一致事务。

## 7. 为什么 targetPrice 放在 watchlist 表？

目标价是“用户对某个商品的关注设置”，不是商品本身属性。不同用户关注同一商品时目标价可以不同，通知开关也可以不同，所以 `target_price`、`notify_enabled`、`last_notified_price` 都属于 `tb_watchlist`。

## 8. 定时任务如何刷新价格？

应用入口使用 `@EnableScheduling` 开启调度。`PriceRefreshTask.refreshActiveProducts` 按 cron 触发，调用 `PriceService.refreshActiveProducts`。Service 按商品 ID 分页扫描 `status = 1` 的商品，对每个商品执行刷新逻辑。单个商品失败最多重试 2 次，最终失败也只记录日志，不中断整个批次。

## 9. 用户量变大后的系统瓶颈在哪里？

可能瓶颈包括：热点商品缓存失效瞬间打爆 MySQL、定时任务扫描商品效率不足、RabbitMQ 消息堆积、通知表查询变慢、Redis 幂等 key 数量增长、单体应用线程池和数据库连接池不足。后续可以通过热点缓存预热、任务分片、MQ 消费者扩容、通知表索引优化、消息监控和水平扩容处理。

## 10. 缓存穿透、击穿、雪崩如何处理？

穿透：商品不存在或失效时写短 TTL 空值缓存。击穿：商品详情和价格回源前尝试 Redis 分布式锁，避免大量请求同时查库。雪崩：正常缓存 TTL 加随机偏移，减少大量 key 同时过期。

## 11. RabbitMQ 重复消费如何处理？

Consumer 收到消息后先构造幂等 key。优先使用 `messageId`，没有时使用用户、商品、目标价、当前价、触发时间等业务字段兜底。通过 Redis `setIfAbsent` 写入成功才继续处理，写入失败说明消息已处理或处理中，直接跳过。

## 12. 如何证明项目真实跑通？

可以从三方面证明。接口层：按 README 的完整链路执行注册、登录、创建商品、添加关注、刷新价格、查询通知。数据库层：查询 `tb_price_history` 有价格变化记录，`tb_notification` 有 `TARGET_PRICE_REACHED` 记录。日志和 MQ 层：日志出现 `Publishing price alert message`、`Received price alert message from queue=price.alert.queue`、`Notification send success`。

## 13. 项目中最难的问题是什么？

最难的是把价格刷新、关注设置、异步消息、通知落库和防重串成一条稳定链路。单独写 CRUD 不难，难点在于边界条件：价格未达到目标价不能发消息，重复刷新不能重复通知，MQ 可能重复消费，Consumer 处理失败要有日志，商品缓存更新后要失效。项目通过 Redis 幂等、业务二次校验、日志和测试覆盖把链路收住。

## 14. 为什么不直接在刷新价格时写通知表？

直接写通知表实现简单，但刷新价格会被通知落库拖慢，也不利于未来扩展短信、邮件等通知渠道。使用 RabbitMQ 后，刷新流程只需要判断条件并发消息，通知处理可以异步消费，后续扩展也更自然。

## 15. 价格刷新为什么用 mock？

当前项目重点是后端链路，不是爬虫或第三方价格采集。`PriceMockUtil` 用于稳定演示价格变化、价格历史、目标价判断和通知链路。后续可以替换为真实采集服务，而不改变关注、通知和 MQ 的主流程设计。

## 16. 如果 Redis 挂了会怎样？

当前实现依赖 Redis 做缓存、锁、限流和幂等。如果 Redis 不可用，相关操作会失败并影响接口。生产环境可以增加降级策略，例如缓存失败时回源 MySQL、幂等失败时保守拒绝或落数据库幂等表，并配合 Redis 高可用部署。

## 17. 通知表为什么还需要 last_notified_price？

Redis 幂等 key 有 TTL，MQ 也可能重复消费。`last_notified_price` 是业务层的最终防线，用来表达“这个关注记录对这个价格已经提醒过”。即使 Redis key 过期，只要同一价格再次到来，也可以跳过重复通知。

## 18. 项目目前有哪些生产级不足？

当前没有死信队列、延迟重试、数据库级通知唯一约束、真实价格采集、Docker Compose、完整压测结果和多通知渠道。它已经适合作为后端学习和面试项目，但生产化还需要补可靠性、可观测性、部署和容量评估。

## 19. 如果 RabbitMQ 消息发出去了，但是 Consumer 没消费成功，你怎么处理？
我不会只依赖 MQ 的重投机制，因为重投可能导致重复消费，所以 Consumer 端必须先做好幂等。然后再配合重试、死信队列和消息状态记录，保证失败消息可追踪、可恢复。

## 20. 如果价格刷新任务同时刷新同一个商品，会有什么问题？你怎么处理？
如果多个价格刷新任务同时刷新同一个商品，可能会出现几个问题。

第一，价格历史可能重复写入。两个任务同时读取旧价格，然后分别生成新价格并写入 tb_price_history，最后会出现重复或顺序混乱的价格记录。

第二，通知可能重复触发。两个任务都判断 currentPrice 小于等于 targetPrice，就可能同时发送 RabbitMQ 消息，导致重复通知风险。

第三，缓存一致性可能受影响。两个任务同时更新价格、删除缓存，可能造成短时间内缓存和数据库状态不一致。

我的处理思路是把商品维度作为并发控制粒度。刷新某个商品前，先基于 productId 尝试获取 Redis 分布式锁，比如 key 是 price_tracker:product:{productId}:lock。拿到锁的任务才执行刷新，没拿到锁的任务直接跳过或稍后重试。锁要设置短 TTL，避免任务异常退出后锁永久不释放。

同时通知链路还有 Redis 入口幂等、Consumer messageId 幂等和 last_notified_price 业务防重，所以即使出现重复刷新，也能降低重复通知风险。
