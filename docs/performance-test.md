# Price Tracker Performance Test Plan

本文档用于第三阶段高并发增强版的联调与压测记录。文档只提供方案、场景和记录模板，不填写未经真实压测验证的 QPS、响应时间或优化百分比。

## 测试目标

- 验证 Redis 缓存对商品详情、商品价格、用户关注列表查询的保护效果。
- 验证缓存穿透、缓存击穿、缓存雪崩保护在压测场景下的行为。
- 验证关注相关写接口在高并发下的限流和幂等行为。
- 验证价格刷新批处理在多商品场景下的稳定性。
- 验证 RabbitMQ 通知消费者幂等和消费失败处理不会造成重复通知或主流程中断。

## 测试前置条件

- MySQL 数据库 `price_tracker` 已创建，Spring Boot 应用已启动并由 Flyway 完成 `src/main/resources/db/migration/` 下的 schema migration。
- Redis 已启动，应用可通过 `application.yml` 中的 Redis 配置连接。
- RabbitMQ 已启动，应用可连接 `price.alert.exchange`、`price.alert.queue` 和 routing key `price.alert`。
- 应用已启动，Knife4j 或接口工具可访问基础接口。
- 已准备可登录测试用户，并获取 JWT Token。
- JMeter 请求已统一添加 `Authorization: Bearer {token}`。
- 压测前清理或记录 Redis 初始状态，避免误判缓存命中。
- 压测前记录 MySQL、Redis、RabbitMQ 和应用日志采集方式。

## JMeter 测试场景

### 场景 1：商品详情缓存命中

- 预热：先请求一次 `/api/products/{id}`，确保商品详情缓存已写入。
- 压测：持续请求同一有效商品详情。
- 观察：平均响应时间、P95、P99、QPS、错误率、Redis 命中情况、MySQL 查询变化。

### 场景 2：不存在商品空值缓存

- 预热：请求一次不存在的商品 ID。
- 压测：持续请求同一不存在商品 ID。
- 观察：是否稳定返回业务错误，MySQL 查询是否被空值缓存降低。

### 场景 3：热点商品击穿保护

- 准备：清理某个热点商品详情和价格缓存。
- 压测：多并发同时请求同一商品详情或商品价格。
- 观察：只有少量请求回源，其他请求命中缓存或收到重建中提示。

### 场景 4：用户关注列表缓存

- 预热：请求 `/api/watchlist/my`。
- 压测：持续查询当前用户关注列表。
- 观察：Redis 命中情况、MySQL 查询变化、分页返回稳定性。

### 场景 5：关注写接口限流和幂等

- 压测：对 `/api/watchlist`、`/api/watchlist/{id}`、`DELETE /api/watchlist/{id}` 分别压测。
- 观察：重复关注是否返回已有记录 ID；超过阈值时是否返回 `request too frequent`。

### 场景 6：手动刷新商品价格与 MQ 通知链路

- 压测：请求 `/api/internal/products/{id}/refresh-price`。
- 观察：商品价格缓存是否失效，价格历史是否写入，达到目标价时是否投递 MQ 消息，消费者幂等是否生效。

### 场景 7：价格刷新批处理

- 触发：执行定时任务或直接调用批量刷新入口。
- 观察：分页批处理日志、单商品失败隔离、重试次数、整体任务耗时。

## 风险与后续优化方向

- Redis 当前承担缓存、分布式锁、限流和幂等 key，后续需要结合监控评估内存、连接数和热点 key 风险。
- 当前缓存命中率需要依赖日志或外部监控统计，后续可补充应用侧指标埋点。
- 当前通知幂等主要依赖 Redis TTL，后续可补充数据库唯一约束或消息幂等表。
- 当前 MQ 链路未引入死信队列和统一重试编排，后续可补充失败消息治理方案。
- 当前压测文档不包含 Docker Compose 环境，后续可补充 MySQL、Redis、RabbitMQ 的一键联调环境。
